# AutoOpt Readiness Handout - 2026-05-10

This handout captures the decisions, findings, and next steps from the chat about getting Universal Spell Check ready for desktop-app speed optimization.

## Original Goal

Identify what needs to happen before the repo is ready to run autonomous speed optimization on the native desktop app.

The target vision is a Karpathy-style auto-research loop:

1. Propose one performance change.
2. Build the app.
3. Run deterministic speed tests.
4. Run correctness checks.
5. Keep the change only if it is faster and still correct.
6. Revert and move on if it fails, regresses, or does not improve enough.

## API Call Count Discovery

The bench makes one OpenAI API call per input per measured or warmup trial:

```text
calls = 20 inputs * (runs + warmup)
```

Call counts:

| Bench config | Calls |
|---|---:|
| `--runs 1 --warmup 0` | 20 |
| `--runs 3 --warmup 1` | 80 |
| `--runs 10 --warmup 2` | 240 |
| `--runs 20 --warmup 3` | 460 |

Original autoopt setting:

```powershell
--runs 20 --warmup 3
```

Problem:

- That costs 460 API calls per bench.
- A 50-iteration run could approach 23,460 API calls if every iteration reaches the bench step.
- This is too expensive for exploratory optimization.

Decision:

- API call budget is now part of the autoopt design.
- The optimization loop should use cheap screening first, not heavy benchmarking every iteration.

## AutoOpt Budget Decision

New default loop:

| Stage | Command flags | Calls | Purpose |
|---|---:|---:|---|
| Screen | `--runs 3 --warmup 1` | 80 | Used every iteration. Cheap enough for exploration. |
| Confirm | `--runs 10 --warmup 2` | 240 | Used only when screen results show a likely win. |

Thresholds:

- Screen must improve median `request_ms` by at least 8%.
- Confirmation must improve median `request_ms` by at least 5%.
- A normal rejected iteration now costs 80 calls instead of 460.

Setup cost:

- 80 calls for screen baseline.
- 240 calls for confirmation baseline.
- 320 calls total before iteration starts.

Reasoning:

- The screen run is cheaper but noisier, so it needs a higher threshold.
- The confirmation run is heavier and should decide whether a change is actually worth committing.

## Future Plan

Before running a real autoopt loop:

1. Make sure the working tree is clean or intentionally committed/stashed.
2. Run a fresh screen baseline:

   ```powershell
   dotnet run --project bench\UniversalSpellCheck.Bench.csproj -c Release -- --runs 3 --warmup 1 --variant baseline-autoopt-screen
   ```

3. Run correctness on the screen baseline:

   ```powershell
   python bench\check_correctness.py <screen-baseline.json>
   ```

4. Run a fresh confirmation baseline:

   ```powershell
   dotnet run --project bench\UniversalSpellCheck.Bench.csproj -c Release -- --runs 10 --warmup 2 --variant baseline-autoopt-confirm
   ```

5. Run correctness on the confirmation baseline:

   ```powershell
   python bench\check_correctness.py <confirmation-baseline.json>
   ```

6. Initialize autoopt state:

   ```text
   bench/autoopt/baseline.json
   bench/autoopt/confirmation-baseline.json
   bench/autoopt/journal.jsonl
   ```

7. Run a short pilot loop first, such as 3-5 iterations.
8. Review commits and journal output before trusting a long run.
9. Only after the pilot works, consider a longer overnight-style run.

## Short Recommendation

The repo is close to ready, but it should not run the old 460-call-per-iteration loop.

The right next move is:

1. Keep the lower-call AutoOpt design.
2. Generate fresh baselines.
3. Run a small pilot loop.
4. Review the journal and commits.
5. Only then scale the iteration count.
